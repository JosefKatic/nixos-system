## Context

Currently `caelestia-shell` is a flake input providing a pre-built quickshell config. Home-manager manages `~/.config` declaratively — files are read-only symlinks from the Nix store. Any runtime write (e.g., matugen outputting a color palette) either gets blocked or clobbered on next `home-manager switch`. This makes a dynamic wallpaper → Material You regen → live reload pipeline impossible without rebuilds.

Four approaches were evaluated for dynamic theming ergonomics:
1. **Keep HM, leave specific dirs unmanaged** — surgical workaround
2. **nix-maid** — HM alternative with mutable-friendly static-dir approach
3. **wrapper-manager wrappers** — bake config paths into wrapped executables, no `~/.config` needed
4. **Drop HM entirely** — replace with NixOS user packages + manual dotfile layer

## Goals / Non-Goals

**Goals:**
- Custom quickshell shell config in-repo (not an upstream flake)
- Wallpaper picker that triggers matugen at runtime
- Live theme reload without rebuilds
- Works on alcedo (desktop) and hirundo (laptop)
- Theme config dirs freely writable by matugen at runtime

**Non-Goals:**
- NixOS specialisations for theming
- GTK/Qt theming integration in this change
- Server hosts

## Decisions

### Decision 1: Config mutability strategy

**Option A — Keep HM, leave theme dirs unmanaged**
Home-manager manages everything except `~/.config/matugen/` and `~/.config/quickshell/colors/`. An activation script creates these dirs and leaves them writable.
- Pros: minimal disruption, HM still handles everything else
- Cons: HM-managed and unmanaged dirs coexist; easy to accidentally let HM clobber a dir on switch; per-app theming still requires rebuilds for anything HM controls

**Option B — nix-maid**
[nix-maid](https://github.com/viperML/nix-maid) replaces HM file management with a "static directory" approach (analogous to NixOS `/etc`): `~/.gitconfig → <static>/.gitconfig → /nix/store/...`. It supports mustache templating resolved at activation time and uses systemd units instead of activation scripts for fast, atomic upgrades. Crucially, files _not_ declared in nix-maid are simply not managed — no clobbering.
- Pros: cleaner model than HM for dotfiles; unmanaged dirs are naturally untouched; simpler than HM overall; avoids `mkOutOfStoreSymlink` hacks
- Cons: young project, less ecosystem coverage than HM; migrating all current HM config is significant work; loses HM's `programs.*` abstraction layer and user service management

**Option C — wrapper-manager wrappers**
[wrapper-manager](https://github.com/viperML/wrapper-manager) creates wrapped executables that self-bundle config via CLI flags and env vars — `~/.config/<app>/` is not needed at all. Example: wrap quickshell with `--config /path/to/store/config.qml`; wrap other apps to point `XDG_CONFIG_HOME` or specific config flags at runtime-writable paths generated by matugen.
- Pros: configs live in Nix store (reproducible), no `~/.config` management at all for wrapped apps; matugen can freely write color tokens to a runtime dir without conflicting
- Cons: not all apps support config-path flags; quickshell specifically may not expose a clean `--config` override for the full shell; partial solution (doesn't help apps that hardcode XDG paths)

**Option D — Drop HM entirely**
Use `users.users.<name>.packages` in NixOS for packages, plain symlinks/scripts for dotfiles. Full freedom, maximum complexity.
- Cons: lose user services, `programs.*` modules, per-user package sets. Too much lost for too little gain.

**Chosen: Option A now + Option B (nix-maid) as explicit follow-up change**

Option A unblocks quickshell + matugen work immediately. The HM boundary is kept explicit in the module (only color output dirs are unmanaged). Option B (nix-maid migration) and Option C (wrappers for specific apps) are recorded as a planned follow-up — migrating to nix-maid would fully solve the "any app config is runtime-mutable" problem, and wrapper-manager can complement it for apps that support config flags. Option D is rejected.

### Decision 2: Quickshell config as in-repo source

Store QML sources under `modules/home/user/desktop/wayland/quickshell/config/` and install via `home.file` (read-only store symlinks). Theme output dirs (`~/.config/quickshell/colors/`, `~/.config/matugen/`) created by activation script, not tracked by HM.

**Alternative considered:** Fork caelestia-shell as a flake — rejected (unnecessary indirection, harder to iterate).

### Decision 3: Wallpaper picker as quickshell component

A quickshell QML component lists images from a configured wallpaper directory. On selection, calls `matugen image <path>` then signals quickshell to hot-reload colors. Wallpaper dir exposed as a HM option (`quickshell.wallpaperDir`, default `~/Pictures/Wallpapers`).

### Decision 4: Impermanence compatibility

Hosts using impermanence must persist `~/.config/matugen/` and `~/.config/quickshell/colors/`. The quickshell module exposes `quickshell.persist = true` which adds these to `home.persistence`. Module warns at eval time if impermanence is detected and the option is unset.

## Risks / Trade-offs

- **HM/non-HM boundary drift**: Unmanaged dirs won't be cleaned up by HM. Mitigation: scope to color output dirs only, document clearly in module.
- **nix-maid migration scope**: Migrating all HM file config to nix-maid is non-trivial. Mitigation: design module boundaries now so HM file management is localized and swappable.
- **wrapper-manager partial coverage**: Some apps don't support config-path overrides. Mitigation: use wrappers only where they cleanly apply; don't force it.
- **First boot colors**: No colors until user picks a wallpaper. Mitigation: activation script runs matugen with a bundled default wallpaper if no color file exists.
- **impermanence hosts**: Easy to miss persistence entries. Mitigation: module eval-time warning.

## Migration Plan

1. Add quickshell module, deploy with `nh home switch .` on alcedo
2. Verify matugen writes colors, quickshell hot-reloads on wallpaper change
3. Enable on hirundo
4. Keep `caelestia-shell` flake input until quickshell shell is verified stable, then remove

**Rollback:** Re-enable caelestia-shell module, disable quickshell module — no data loss.
